# Part 4: Technical Communication

I chose PR #1172 over all other PRs because I have too built a local RAG engine (with vector database) and dealt with the similar problem that this PR solves (i.e., generating an embedding and then generating an answer are two different jobs and relying on the same provider config for both of them can create problems). I was using Ollama for embeddings while doing my own testing for answer generation, and there was no neat way to separate the configs to make sense of my setups (which was a messy situation for me). Therefore, when I was reading this PR, it caugh my attention and i decided to pick that one.

On the technical side, this PR covers technologies with which I have experience in, in building FastAPI applications and designing REST APIs, so it made sense to me as I was reading about how the EmbeddingConfig class is structured as a Pydantic model. I use that same pattern in my own projects to validate configs. I also have experience in vector search (HNSW, KD-Tree) using FAISS-adjacent solutions, so I understood the dimension auto-fill fix as it applied to FAISS right away. in addition to that, I also noticed that the problem with using 1536 as a hardcoded value would cause an issue when switching between Gemini and Ollama right away.

I do not worry too much about persisted FAISS index scenarios. This PR only fills the dimensions automatically from the active embedding provider (which is good for new indexes). But, in production, a team has already persisted their FAISS indexes to disk after incurring the expense of doing the indexing (this can result in millions of document chunks). If the original index was built using OpenAI embeddings and dimensions of 1536, then the team switches to Gemini embeddings and dimensions of 768, the result of that index loading could either fail with a silent dimension mismatch, or worse result in incorrect retrievals with no error message from FAISS. FAISS does not always validate dimensions while loading, it only tries to operate on the math of the numbers. This type of bug would present itself at runtime when producing garbage results from queries and would not be evident from running the tests.

My method of going about this is to create a small file of metadata stored with each of the persisted indices that lists which embedding configuration was used to build each index including the provider, model name, and dimensions. When loading an index, the engine reads from this file, compares the metadata to the current configuration, and runs an error message before running any other code. If the configuration and metadata do not match, it produces an immediately visible error. I did something similar in my project where I retained a set of parameters regarding how an index was built in the same location as the serialized-encoded vectors to avoid this type of silent corruption. The additional code is relatively simple (~20 lines), and yet provides protection from a possibly production issue.
